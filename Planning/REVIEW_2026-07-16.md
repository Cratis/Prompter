## Fixed in this pass (branch `review/2026-07-16-followups`)

- **[High] `UserHash` did not deliver the D-8 pseudonymization it promised** — `Source/Discord/UserHash.cs`.
  Bare `SHA256("prompter:{snowflake}")` over a public, enumerable id, in a public repo, is reversible by anyone
  holding the interaction log (hash the roster, match rows). Switched to **HMAC-SHA256 keyed** with a secret
  (`Cratis:Prompter:Discord:UserHashKey`), **required at startup whenever a Discord token is set**. Adds
  `for_UserHash` specs (determinism, keyed-ness, cross-user distinctness). Documented in the README config
  table + deployment secret list.
- **[High] Lexical retrieval arm dropped its top matches** — `Source/Retrieval/Passages.cs`. The `lexical` CTE
  ended in `LIMIT` with no statement-level `ORDER BY`, so `RANK()` was correct but the `LIMIT` kept an
  arbitrary 20 rows (undefined output order) whenever >20 chunks matched — silently discarding the best BM25
  passages and undercutting D-4. Added `ORDER BY ts_rank_cd(...) DESC` before the `LIMIT`, mirroring the
  semantic arm. Not spec-coverable without a live pgvector DB; will be exercised by the M2.2 calibration run.

## Open findings (ranked)

### Retrieval / answering quality

- **[High, tracked as P-07] Refusal threshold is on the raw RRF scale.** `Answering:MinScore = 0.02` vs an RRF
  max ≈ 0.033 (`RrfConstant = 60`): a passage present in only one candidate list maxes at 1/61 ≈ 0.0164 < 0.02,
  so the default effectively demands the top passage appear in *both* the semantic and lexical top-20 lists.
  This systematically refuses keyword-free paraphrases (the semantic half's whole point) and can still answer
  lexically-overlapping out-of-scope near-misses. The `AskOutput` spec even fixtures an impossible score
  (`0.21`), confirming the default was never validated against real magnitudes. Resolve during P-07 calibration;
  consider gating on an absolute cosine-distance floor rather than the raw fused score.
  (`Source/AnsweringOptions.cs:19`, `Source/Answering/Answers.cs:34-37`)
- **[Medium] Citations are retrieval-derived, not from the model's answer.** The top-4 retrieved pages are
  attached regardless of what the model cited; nothing parses the `[n]` markers the system prompt requests. An
  answer grounded in excerpt [7] shows sources [1]–[4], and the `[n]` in the text don't map to the "Sources"
  list. Also makes the eval's "citation accuracy" trivially always-true.
  (`Source/Answering/Answers.cs:60`, `Source/Answering/SystemPrompt.cs:27`)
- **[Low] Voyage `input_type` omitted** — both corpus and query embed identically, forgoing Voyage's
  asymmetric query/document optimization. (`Source/Embeddings/VoyageEmbeddings.cs`)
- **[Low] Prompt caching is inert** — the ~300-token system prompt is below the model's minimum cacheable
  size, so `WithCacheControl` is a no-op (honestly documented; the D-5 benefit isn't realized yet).

### Ingestion / storage robustness

- **[Medium] Content hash excludes title + heading path.** `MarkdownChunker` hashes only the body, but the
  Indexer embeds `title — headingPath\n\n content` and stores `title`/`heading_path`. A heading/title rename
  with an unchanged body is detected as "unchanged" → stale embedding + wrong citation title persist. Hash the
  same string that is embedded. (`Source/Ingestion/MarkdownChunker.cs:49` vs `Source/Ingestion/Indexer.cs:81`)
- **[Medium] `Voyage:Dimensions` is decoupled from the hardcoded `vector(1024)` schema.** Setting it to
  anything but 1024 fails every upsert at runtime with no fast-fail. Add a startup `Validate` (or make it a
  constant). (`Source/VoyageOptions.cs:29`, `Source/Storage/Migrations/v1_0_0.sql:10`)
- **[Medium] `EnsureSchema` migration apply has no advisory lock** (check-then-act). Two overlapping starts
  (redeploy/replica) both apply the same migration; the loser's version-`INSERT` hits the PK and the pod fails
  to boot. Wrap in `pg_advisory_lock`, or make the version insert `ON CONFLICT (version) DO NOTHING`.
  (`Source/Storage/Chunks.cs:25-36,109-120`)
- **[Medium] Retry net misses status-less transient failures.** Only 429/5xx retry; a TCP reset / DNS blip
  (`HttpRequestException` with null status) or a timeout isn't retried — common on long index runs. Treat null
  status as transient; consider honoring `Retry-After` and adding jitter.
  (`Source/Embeddings/EmbeddingRetry.cs:21`, `Source/Embeddings/ResilientEmbeddingGenerator.cs:41`)
- **[Low] `VoyageEmbeddings` response is never disposed**, and there is **no count/index validation** on the
  positional embedding mapping — a missing middle item would shift every subsequent embedding by one and
  corrupt upserts before throwing. (`Source/Embeddings/VoyageEmbeddings.cs:28-36`)
- **[Low] `RemoveAllExcept(∅)` deletes the whole table**; the empty-crawl guard lives only in the caller
  (`Indexer.Run`). Short-circuit inside the method too (defense in depth). (`Source/Storage/Chunks.cs:81-87`)
- **[Low] RRF `score` is `numeric` read via `GetDouble`** — cast `::double precision` in SQL to be robust
  across Npgsql versions. No ANN index on `embedding` (fine at this corpus size).

### Discord

- **[Medium — verify on test server] Feedback button routing is unverified under NetCord beta.11.** A
  parameterless `Vote()` is bound to a 3-segment custom id (`feedback:up:42`); the code asserts it routes by
  prefix but nothing confirms it. If wrong, it is a silent 100% feedback loss. Covered by the existing
  token-gated live-verification checklist. (`Source/Discord/Feedback.cs:31-32`)
- **[Medium] `DiscordAnswers.Format` truncates body + sources together**, so long `/ask` and forum answers
  silently drop the "Sources:" line and can cut a `<url>` mid-string, breaking the angle-bracket embed
  suppression. The `Split` path (mentions) handles this correctly; the surfaces disagree. Reserve room for the
  sources line and truncate only the body (reuse the `Split` logic). (`Source/Discord/DiscordAnswers.cs:29-35`)
- **[Low] `RateLimiter` buckets are never evicted** — slow unbounded growth over uptime (concurrency itself is
  correct). Sweep full-capacity buckets on a TTL. (`Source/Discord/RateLimiter.cs`)
- **[Low] `Ask.Handle` pre-ack calls sit outside the try** — an ack/defer failure escapes the handler rather
  than being logged (NetCord's outer catch still prevents a crash). (`Source/Discord/Ask.cs:54-63`)
- **[Low] American-English convention** — `Acknowledgement` → `Acknowledgment`. (`Source/Discord/Feedback.cs:24`)

### Operations / config / eval

- **[Medium] Typed `HttpClient`s captured by singletons defeat handler rotation** (already the documented
  known item in the handover) — `VoyageEmbeddings`/`DocsSite` are pinned for process lifetime, so DNS/endpoint
  changes aren't picked up until restart. Low real impact single-instance. (`Source/Hosting/HostBuilderExtensions.cs:54-71`)
- **[Medium] Eval gate has a blind spot + placeholder baseline.** The enforced metrics omit in-scope
  answer-rate, and `baseline.json` is a documented placeholder — so the P-19 gate wouldn't catch the
  over-refusal that the threshold finding makes the likely regression, and enforces nothing real until
  regenerated from a keyed run. (`Eval/EvalReport.cs:42-49`, `Eval/baseline.json`)
- **[Low] Startup validation gaps** — `AnswerTimeoutSeconds` (0/negative → every answer apologizes) and
  `Discord.Token` (empty → late gateway failure) aren't in the `ValidateOnStart` chain.
- **[Low] Reindex fire-and-forget is uncancellable on shutdown** — pass `ApplicationStopping` into
  `indexer.Run(token)`. `ReindexAuth` leaks the secret *length* via `FixedTimeEquals` (hash both sides first).
  (`Source/Operations/PrompterEndpoints.cs:78-93`, `Source/Operations/ReindexAuth.cs:35-38`)

### Prompt-injection posture

- **[Medium] Question + passages share one user turn with spoofable plain-text labels** — a user can inject a
  fake `Documentation excerpts:` block. Blast radius is limited (citations come from retrieval, not from
  parsing the answer, so a citation URL can't be forged), but ungrounded output can be coaxed. Put the question
  in a separate user message and/or wrap excerpts in a non-spoofable boundary. (`Source/Answering/Answers.cs:71-88`)
